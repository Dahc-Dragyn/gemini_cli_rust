Software Test Plan: gemini_cli_rust1. Objectives & ScopeObjective: Ensure the CLI operates reliably as a zero-dependency appliance, safely handles missing or invalid API keys, correctly formats terminal output (including streams), and adheres to standard UNIX exit codes.In-Scope:CLI argument parsing (clap).JSON serialization and deserialization (serde).HTTP request formatting and error handling.stdin pipe detection and ingestion.Standard output (stdout) and streaming formatting.Out-of-Scope:Performance testing of Google's Gemini API infrastructure.Cross-platform terminal emulator rendering quirks (e.g., specific ANSI color support on legacy Windows CMD).2. Testing Frameworks & ToolsTo keep the project lightweight, we will use the native Rust ecosystem:cargo test: The built-in test runner for Unit and Integration tests.assert_cmd: A crate specifically designed to test CLI binaries (allows us to simulate running gemini-rs in the terminal and assert the exit codes/stdout).mockito or wiremock: Crates for spinning up a local mock HTTP server. This allows us to test API responses (including streaming chunks) without actually hitting Google's servers or spending API credits during CI/CD.3. Test Phases & StrategyPhase 1: Unit Testing (Internal Logic)Focus: Testing individual functions and structs in isolation.CLI Parsing: Verify clap correctly parses valid flags (--stream, --model) and rejects invalid ones.JSON Serialization: Ensure the GenerateContentRequest struct serializes into the exact JSON format expected by the Gemini API.JSON Deserialization: Provide mock JSON strings of Gemini API responses (both success and error payloads) and ensure serde parses them into our Rust structs without panicking.Prompt Construction: If piping stdin, verify the function that concatenates the stdin buffer with the user's CLI argument formats the final string correctly.Phase 2: Integration Testing (Mocked Network)Focus: Testing the flow from CLI input -> HTTP Client -> stdout, without hitting the real internet.The Happy Path (One-Shot): Use assert_cmd to run the binary, route reqwest to a mockito local server returning a 200 OK JSON response, and assert that the output matches the expected text and exits with code 0.The Happy Path (Streaming): Mock a Server-Sent Events (SSE) stream and verify the CLI flushes chunks to stdout sequentially.Auth Failure (401/403): Mock an invalid API key response. Assert the CLI does not panic, prints a friendly error message to stderr, and exits with code 1.Rate Limiting (429): Mock a Too Many Requests response. Assert the CLI exits gracefully with code 1 and a backoff warning.Network Timeout: Configure the HTTP client with a 1-second timeout, delay the mock server response by 2 seconds, and assert the CLI handles the timeout gracefully without hanging indefinitely.Phase 3: System / End-to-End (E2E) TestingFocus: Testing the compiled release binary against the live Google API.(Note: These tests should be hidden behind a feature flag like #[ignore] or run only when a specific environment variable like RUN_LIVE_TESTS=1 is set, to avoid spamming the API).Live One-Shot: Execute gemini-rs ask "Say the word 'Test'" and assert the output contains "Test".Live Stream: Execute gemini-rs ask "Count to 5" --stream and verify the stream completes successfully.Pipe Execution: Simulate echo "Hello" | gemini-rs ask "Repeat this" and assert the model receives and repeats the piped context.4.
